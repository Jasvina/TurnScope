# Codex App-Server to TurnScope Mapping

## Status

Bootstrap mapping for the first real runtime adapter.

## Primary source

This mapping is based on the official Codex app-server README:

- `https://github.com/openai/codex/blob/main/codex-rs/app-server/README.md`

Key parts used:

- turn lifecycle notifications
- item lifecycle notifications
- command execution approvals
- file change approvals
- dynamic tool calls
- command execution, fileChange, mcpToolCall, collabToolCall item shapes

## Design choice: thread id as session id

Codex app-server exposes a durable `threadId` and per-turn `turnId`.

For TurnScope v0.1, the adapter maps:

- `threadId` -> `session_id`
- `turnId` -> `turn_id`

Rationale:

- the thread is the closest durable unit to a TurnScope session
- it is stable across multiple turn notifications
- it lets us reopen and group stored history without inventing a synthetic session id

## Canonical mapping table

| Codex notification or request | Condition | TurnScope event |
| --- | --- | --- |
| `turn/started` | always | `turn.started` |
| `turn/completed` | always | `turn.finished` |
| `item/started` | `item.type = commandExecution` | `shell.started` |
| `item/commandExecution/outputDelta` | always | `shell.output` |
| `item/completed` | `item.type = commandExecution` | `shell.finished` |
| `item/completed` | `item.type = fileChange` | `file.changed` |
| `item/started` | `item.type = mcpToolCall` | `tool.called` |
| `item/completed` | `item.type = mcpToolCall` | `tool.finished` |
| `item/started` | `item.type = dynamicToolCall` | `tool.called` |
| `item/completed` | `item.type = dynamicToolCall` | `tool.finished` |
| `item/started` | `item.type = webSearch` | `tool.called` |
| `item/completed` | `item.type = webSearch` | `tool.finished` |
| `item/completed` | `item.type = collabToolCall` and `tool = spawn_agent` | `subagent.spawned` |
| `item/commandExecution/requestApproval` | always | `approval.requested` |
| `item/fileChange/requestApproval` | always | `approval.requested` |
| `item/permissions/requestApproval` | always | `approval.requested` |
| `serverRequest/resolved` | request id previously seen in approval request | `approval.resolved` |
| `error` | always | `error.raised` |

## Explicit non-mappings for now

These messages are not mapped yet in the bootstrap adapter:

- `thread/tokenUsage/updated`
- `turn/plan/updated`
- `turn/diff/updated`
- `model/rerouted`
- audio or realtime thread notifications
- unstable auto-approval review notifications

Reasons:

- they deserve either their own canonical event types or richer attributes
- forcing them into the current v0.1 envelope would hide meaning

## Timestamp strategy

Codex notifications in raw JSONL capture do not always carry a canonical timestamp field.

The adapter uses this priority order:

1. top-level `receivedAt`
2. top-level `timestamp`
3. known fields on `turn` or `item` payloads such as `startedAt`, `completedAt`, or `updatedAt`
4. synthetic monotonic fallback timestamps generated by the adapter

## Approval behavior notes

Approvals in app-server are not ordinary notifications only; they may arrive as server-initiated JSON-RPC requests with an `id`.

The adapter preserves that `id` as the approval request handle and emits:

- `approval.requested` when the request first appears
- `approval.resolved` when `serverRequest/resolved` confirms cleanup or completion

The final outcome of the action still comes from the associated `item/completed` event.

## Bootstrap scope

The goal of this mapping is not perfect completeness. The goal is to turn real Codex app-server captures into useful TurnScope timelines with:

- turns
- shell activity
- tool activity
- file changes
- approvals
- subagent spawns
- errors

That is enough to drive the next UI iteration.
